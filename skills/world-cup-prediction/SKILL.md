# World Cup Prediction

Use this skill when the user asks for:

- a World Cup match prediction
- tomorrow's or today's World Cup fixtures
- one or more fixture IDs
- a refreshed prematch prediction after data changes
- a formal or semi-formal prediction summary

## Core rules

1. Prediction numbers come from repo scripts, not LLM freewriting.
2. Run `world-cup-predict` or the script wrapper before giving probabilities.
3. Structured JSON comes first; natural-language explanation comes second.
4. The LLM may explain gaps, but may not invent probabilities, odds, BTTS values, stake sizing, or market conclusions.
5. If a field is missing or unsupported, report the gap explicitly.

## Workflow

1. Resolve target fixtures from `outputs/research_local.db` by fixture id or local date.
2. For natural-language requests like "predict tomorrow's matches", convert the date to `--local-date YYYY-MM-DD` in the user's local context.
3. Run the prediction pipeline:

```powershell
world-cup-predict `
  --db-path outputs\research_local.db `
  --local-date YYYY-MM-DD
```

For exact fixtures:

```powershell
world-cup-predict `
  --db-path outputs\research_local.db `
  --fixture-id fixture_wc2026_66456916
```

4. The command may run provider backfill first, then builds predictions through `PreMatchResearchScoringService`.
5. Read the returned JSON and preserve the script-produced values.
6. If `status` is `partial` or `failed`, explain the blocker from `gaps`, `backfill_error`, `coverage`, or `source`.

## No-backfill mode

Use this only when the user explicitly asks to use existing local data:

```powershell
world-cup-predict `
  --db-path outputs\research_local.db `
  --local-date YYYY-MM-DD `
  --no-backfill
```

## Output contract

The command returns:

```text
schema_version: world_cup_prediction.v1
status: ok | partial | failed
request
source
backfill_error
predictions: array
```

Each prediction must include:

- `fixture_id`
- `match_time_beijing`
- `home_team`
- `away_team`
- `data_status`
- `probabilities.home_win`
- `probabilities.draw`
- `probabilities.away_win`
- `probabilities.over_2_5`
- `probabilities.under_2_5`
- `risk.level`
- `risk.confidence`
- `coverage`
- `calibration`
- `vip.main_pick`
- `vip.secondary_pick`
- `vip.risk_level`
- `sources`
- `gaps`

BTTS, scorelines, capital allocation, and risk/reward are not generated by the open algorithm unless the script returns them. If they are absent or null, state that they are unsupported by the current open model.

## Final response order

1. JSON status and data status.
2. Per-fixture probability summary.
3. Risk and coverage note.
4. Missing or unsupported fields.
5. Sources/provider note.

## Guardrails

- Do not guess fixtures when the DB can resolve them.
- Do not present odds or predictions as a wagering guarantee.
- Do not change script-produced probabilities.
- Do not hide `partial` coverage.
- Do not silently drop unsupported BTTS or staking fields.
